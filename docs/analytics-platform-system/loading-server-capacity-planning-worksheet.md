# <a name="loading-server-capacity-planning-worksheet-for-analytics-platform-system"></a>Загрузка журнала планирования емкости сервера для система платформы аналитики
Этот лист планирования емкости поможет вам определить требования к загрузка сервера для загрузки данных в SQL Server PDW. Используется для создания плана для приобретения и подготовки существующих загрузке серверов.  
  
## <a name="worksheet-notes"></a>Заметки о листа
  
1.  Этот лист относится к серверам, будет загрузка данных с помощью **dwloader** загрузки средство командной строки.  
  
2.  Для загрузки данных инструментом служб Integration Services или третьей стороной загрузки, требования может меняться в зависимости от различия в процессе загрузки.  
  
3.  Применить большинство требований к загрузке файлов либо сжатых и несжатых данных. любые изменения в требованиях отмечены полужирным шрифтом.  
  
## <a name="clipboardmediaclipboard-iconpng-clipboard-capacity-planning-worksheet"></a>![Буфер обмена](media/clipboard-icon.png "буфер обмена") листа планирования емкости  
  
Печать листа и наполнить требованиями.  
  
|Компонент|Требование|Заполните этот столбец с требованиями|Рекомендации|  
|-------------|---------------|--------------------------------------------------|-------------------|  
|Память|Максимально число байтов, которые вы собираетесь сохранить на сервере загрузки на любой период времени.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|Чтобы определить требования к хранилищу, выясните, какой объем данных будет храниться на сервере загрузки на любой период времени.  Требования к емкости, нагрузки только для файлов; операционная система и загрузки файлов должен находиться на разных дисковых массивов.<br /><br />Например: Если планируется загрузить 100 ГБ данных с диска 3 раза в день, но не удаляет файлы данных до конца недели, то вы должны минимальное 2.1 ТБ для хранения файлов данных. Рекомендуется, чтобы он осторожен и о получении 30% больше хранилища для учетной записи для отклонения и роста.  Например лучше 2.73 ТБ дискового пространства.|  
|Скорость загрузки|Максимально число байтов в час для данных для загрузки в PDW.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|Это является приблизительным. При вычислении это требование, предполагается, что файлы уже находятся на сервере загрузки и другие условия нагрузки, как можно лучше.<br /><br />Например: не нужно учитывать сжимаемости данных, так как dwloader всегда отправляет несжатых данных PDW. Не нужно учитывать преобразования типов данных и размер целевой таблицы.|  
|Сеть|Тип сетевого подключения.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|Определите наилучший тип сетевого подключения для требований скорость загрузки.<br /><br />Например: InfiniBand или 10 Гбит Ethernet предоставляет ставки оптимальной загрузки. Ethernet 1 Гбит ограничит скорости загрузки 360 ГБ в час или меньше.|  
|Ввод-вывод|Байт в час для операций чтения и записи.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|Для загрузки данных, dwloader необходимо прочитать все данные с диска перед его отправкой в PDW.<br /><br />Каждый сервер загрузки не удалось загрузить данные быстрее, чем устройство может получать данные из всех источников для загрузки. Чтобы сократить затраты, планирование операций ввода-вывода, емкость для загрузки, чтобы он не превышает емкость устройства чтения.<br /><br />Например:<br />PDW получает и загружает данные в это устройство 1 стойка с максимальной скоростью 1,8 ТБ в час. Для устройства с помощью двух или более стоек коэффициент максимальной нагрузки — 3,6 ТБ в час.<br /><br />Если планируется загрузить из нескольких серверов загрузки, в то же время требования ввода-вывода для каждого сервера загрузки будет меньше, чем когда один сервер выполняет загрузку.<br /><br />Например: один сервер загрузки можно загрузить более 1,8 ТБ в час для 1-для монтажа в стойку устройства. Два сервера загрузки каждого одновременно загрузить 900 ГБ в час в устройство стойка 1. Более высокие уровни параллелизма может снизить эффективность и максимальной пропускной способности.<br /><br />Ресурсы ввода-вывода принимая в учетной записи всех операций ввода-вывода, происходящих на сервере загрузки. Если загрузка сервера имеет другой трафик ввода-вывода в дополнение к загрузке данных, как получение файлов данных с сервера ETL, увеличит требования ввода-вывода.<br /><br />Сжатых данных ввода-вывода требования зависят от степень сжатия данных. dwloader считывает сжатые данные и затем распаковывает его перед отправкой в PDW. Чем выше степень сжатия, тем меньше данных загрузка сервера потребуется чтения с диска.<br /><br />Например: Если частота необходимые нагрузки 1,8 ТБ в час и данные хранятся на сервере загрузки с использованием сжатия 2:1, то загрузка сервера требуется только считать 900 ГБ в час с диска вместо 1,8 ТБ. Коэффициент сжатия 3:1 означает, что загрузка сервера, должен считывать 600 ГБ в час с диска.|  
|ЦП|Число процессоров.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|Для загрузки несжатых данных, dwloader не является приложением процессор.  Как минимум рекомендуется использовать двухпроцессорный сервер недавно производства.<br /><br />Для загрузки сжатых данных, необходимо достаточное количество ЦП, чтобы распаковать данные перед их отправкой PDW. dwloader можно запустить 10 активных потоков одновременно. Если планируется одновременно загрузить 10 сжатых файлов, корпорация Майкрософт рекомендует, что сервер имеет по меньшей мере 10-ядерный ЦП или двумя 6-ядерными процессорами.|  
|ОЗУ|Загружает ГБ памяти, который позволяет Windows для кэширования файлов во время.|![Значок карандаша](media/pencil-icon.png "значок карандаша")|dwloader использует очень мало оперативной памяти на сервере загрузки. Для повышения производительности Windows использует память для кэширования файлов загрузки после считывания их с диска.<br /><br />Чтобы определить требования к ОЗУ, см. установке Windows Server и требования приложения сторонних производителей. Рекомендуется не менее 32 ГБ, если у вас требования из других источников.<br /><br />Сжатых данных быстрее ОЗУ полезно, так как это повышает скорость распаковки.|
