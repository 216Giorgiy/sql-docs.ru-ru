# <a name="securing-rds-applications"></a>Защита приложений служб удаленных рабочих СТОЛОВ
Этот раздел содержит сведения о безопасности для RDS.  
  
> [!IMPORTANT]
>  Начиная с Windows 8 и Windows Server 2012, серверные компоненты служб удаленных рабочих СТОЛОВ больше не включаются в операционной системе Windows (в разделе Windows 8 и [руководство по Windows Server 2012 совместимости](https://www.microsoft.com/en-us/download/details.aspx?id=27416) для получения дополнительных сведений). Клиентские компоненты служб удаленных рабочих СТОЛОВ будут удалены в будущих версиях Windows. Избегайте использования этого компонента в новых разработках и запланируйте изменение существующих приложений, в которых он применяется. Приложения, использующие служб удаленных рабочих СТОЛОВ необходимо перенести в [службы данных WCF](http://go.microsoft.com/fwlink/?LinkId=199565).  
  
## <a name="microsoft-internet-explorer-security-issues"></a>Проблемы безопасности Microsoft Internet Explorer  
 Благодаря новым улучшениям безопасности добавлена в Microsoft Internet Explorer некоторые объекты ADO и RDS ограничены только в среде «безопасной» режиме. Это необходимо учитывать эти проблемы, включая другой зоны, уровни безопасности, строгие поведение, небезопасных операций и пользовательские параметры безопасности.  
  
## <a name="security-and-your-web-server"></a>Безопасность и веб-сервера  
 Если вы используете [RDSServer.DataFactory](../../../ado/reference/rds-api/datafactory-object-rdsserver.md) объекта Internet веб-сервера, помните, что при этом создается потенциальную угрозу безопасности. Внешние пользователи, получившие допустимый источник данных (DSN), идентификатор пользователя и пароль написать страниц для отправки всех запросов к источнику данных. Если требуется более ограниченный доступ к источнику данных, уже отменить регистрацию и удалить **RDSServer.DataFactory** (msadcf.dll) и вместо этого используйте настраиваемые бизнес-объекты с жестко запросами.  
  
 Дополнительные сведения о об обеспечении безопасности с помощью объекта RDSServer.DataFactory см. на безопасности веб-узел корпорации Майкрософт по безопасности MS99-025.  
  
## <a name="client-impersonation-and-security"></a>Олицетворение клиента и безопасность  
 Если **пароль проверки подлинности** вашего веб-сервера IIS является свойство для проверки подлинности Windows NT Challenge/Response (для Windows NT 4.0) или встроенная проверка подлинности Windows (для Windows 2000), то не бизнес-объектов вызывается в контексте безопасности клиента. Это новая функция в 1,5 служб удаленных рабочих СТОЛОВ, позволяющий олицетворение клиента по протоколу HTTP. При работе в этом режиме, имя входа на веб-сервер (IIS) не является анонимным, но использует идентификатор пользователя и пароль, запускающаяся на клиентском компьютере. Если ODBC DSN настроены так, чтобы использовать доверительное соединение, затем доступ к базам данных, например SQL Server, также происходит в контексте безопасности клиента. Но это работает, только если база данных находится на том же компьютере, что и службы IIS; учетные данные клиента не могут переноситься на другой компьютер.  
  
 Например, клиент, John Doe с userid = «JohnD» и пароль = «секрет» вошел на клиентский компьютер. Он запускает приложение на основе браузера, которым нужен доступ к **RDSServer.DataFactory** объект для создания [записей](../../../ado/reference/ado-api/recordset-object-ado.md) , выполнив запрос SQL на компьютере значение «MyServer» под управлением служб IIS. MyServer, система Windows NT Server 4.0 настроена на использование проверки подлинности Windows NT вызов-ответ и его ODBC DSN имеет «Использовать доверенные соединения» выбран сервер также содержит источник данных SQL Server. При получении запроса на веб-сервере, клиент запрашивает идентификатор пользователя и пароль. Таким образом, запрос регистрируется на сервере MyServer как поступающие от «JohnD» / «Секретные» вместо IUSER_MyServer (который используется по умолчанию, если включена анонимная проверка подлинности пароля). Аналогично для входа на SQL Server, «JohnD» / «Секретный» используется.  
  
 Следовательно режим проверки подлинности IIS Windows NT Challenge/Response позволяет HTML-страниц, создаваемых без запроса пользователя явно пользователя сведения ID и пароль, необходимые для входа в базу данных. При использовании обычной проверки подлинности служб IIS были затем это также потребуется.  
  
## <a name="password-authentication"></a>Проверка пароля  
 Служб удаленных рабочих СТОЛОВ может взаимодействовать с веб-сервер IIS под управлением в одном из трех режимов проверки подлинности пароль: анонимный, обычная, или проверку подлинности NT Challenge/Response (называемых встроенную проверку подлинности Windows в Windows 2000). Эти параметры определяют, как веб-сервер управляет доступом через него, например требование, что клиентский компьютер права явный доступ на сервере веб-NT.
