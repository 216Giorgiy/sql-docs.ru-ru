# <a name="high-availability-and-disaster-recovery-for-master-data-services"></a>Высокий уровень доступности и аварийного восстановления для служб Master Data Services

**Аннотация:** в этой статье описывается решение с Master Data Service (MDS), размещенное в конфигурации группы доступности AlwaysOn. В статье рассказывается, как установить и настроить SQL 2016 Master Data Services в группе доступности SQL 2016 AlwaysOn. Основная цель этого решения — обеспечить более высокий уровень доступности и улучшить аварийное восстановление для серверных данных MDS, размещенных в базе данных SQL Server.

## <a name="introduction"></a>Введение


В этой статье описывается решение образца службы данных (MDS), размещен в конфигурации группы доступности AlwaysOn. Статья описывает, как для установки и настройки SQL 2016 MDS для группы доступности AlwaysOn SQL 2016 (AG). Основная цель этого решения — обеспечить более высокий уровень доступности и улучшить аварийное восстановление для серверных данных MDS, размещенных в базе данных SQL Server.

Чтобы реализовать решение, необходимо выполнить следующие задачи, описанные в данной статье.

1.  [Установите и настройте копирование отказоустойчивого кластера Windows Server (WSFC)](#windows-server-failover-cluster-wsfc).

2.  [Настройка группы Доступности AlwaysOn](#sql-server-alwayson-availability-group).

3.  [Настройка служб MDS на узле WSFC](#configure-mds-to-run-on-an-wsfc-node).

Выше разделах кратко накладывает технологий, следуют инструкциям. Подробные сведения о технологиях см. в статье в документах, ссылки для каждого раздела.

Это решение, описанное в этой статье построено на основе Доступности AlwaysOn SQL Server, в котором каждая база данных имеет несколько реплик синхронным или асинхронным. Только одна реплика принимает транзакции (принимает запросы пользователей). Это первичная реплика.

Каждая реплика имеет свое собственное хранилище, поэтому в этом решении нет централизованное общего хранилища. При сбое программного обеспечения или сбоя оборудования, влияющие на первичной реплике первичной возможен переход синхронным или асинхронным реплику, либо автоматически или вручную на основе конфигурации и ситуаций. Это гарантирует высокий уровень доступности базы данных с минимальным прерывания для пользователей.

Асинхронные реплики, обычно размещаются на центр обработки данных, удаленная из центра обработки данных первичной реплики. В случае сценариев аварийного первичной возможен переход на другой центр обработки данных. Это гарантирует аварийного восстановления базы данных.

В целях демонстрации решения, описанные в этой статье использует следующие версии программного обеспечения. Более старые версии должны работать так же, потенциально незначительные отличия.

-   Windows Server 2012 R2 с сервера отказоустойчивого кластера

-   SQL Server 2016 с помощью функции Master Data Service

Кроме того, решение использует две виртуальные машины **MDS HA1** и **MDS HA2**для размещения двух реплик. Поскольку он поддерживается в SQL Server AlwaysOn AG, MDS не ограничивает сколько реплик, которые можно использовать.

В этой статье предполагается наличие базовых знаний о Windows Server, отказоустойчивый кластер Windows Server, SQL Server AlwaysOn и SQL Server MDS.

## <a name="what-is-not-covered"></a>Что не охватывает

В этом документе описывается следующее:

-   Внесение IIS, веб-сервере в образец службы данных пользовательского интерфейса, высокой доступности и возможностью восстановления после аварии. MDS никак не любого определенного требования в IIS, поэтому стандартные технологии для обеспечения высокой доступности службы IIS и балансировки нагрузки можно здесь также работать.

-   Инструкции по использованию SQL Server AlwaysOn (FCI) отказоустойчивого кластера для поддержки высокой доступности (HA) на сервере MDS. Отказоустойчивая кластеризация SQL Server другого решения высокого уровня ДОСТУПНОСТИ и официально поддерживается SQL Server, и работать с MDS.

-   Инструкции по использованию гибридное решение отказоустойчивого кластера SQL Server (FCI) и группы Доступности AlwaysOn для поддержки высокой ДОСТУПНОСТИ на сервере MDS. Гибридное решение работает с MDS.

## <a name="design-consideration"></a>При проектировании

Рисунок 1 демонстрирует типичную конфигурацию, используемую в основном группы Доступности AlwaysOn. Существуют две реплики с синхронной фиксацией связь в первичного центра обработки данных, и обе реплики имеют право ГОЛОСА. Используется главным образом для повышения НАДЕЖНОСТИ в случае сбоя первичной реплики.

В центре обработки данных аварийного восстановления нет вторичную реплику с асинхронной фиксацией связь с первичной. Этот центр обработки данных обычно находится в географическом регионе, отличный от основного центра обработки данных. Вторичная реплика не имеет прав ГОЛОСА.

Эта конфигурация используется для обеспечения восстановления в случае основной ЦОД после аварии, например пожар, наводнение, и т. д. Конфигурации позволяет достичь обоих высокого уровня ДОСТУПНОСТИ и аварийного восстановления с относительно низкой цене.

## <a name="windows-server-failover-cluster-wsfc"></a>Отказоустойчивый кластер Windows Server (WSFC)

В этом разделе рассматриваются следующие задачи.

1.  [Установить компонент отказоустойчивого кластера Windows](#install-failover-cluster-feature).

2.  [Создание отказоустойчивого кластера Windows Server](#create-a-windows-server-failover-cluster).

Как показано на рис. 1 в предыдущем разделе, описанных в этой статье решение содержит отказоустойчивого кластера (WSFC). Необходимо задать WSFC, поскольку SQL AlwaysOn зависит от WFSC для обнаружения сбоя и обеспечения отказоустойчивости.

WSFC — это компонент, чтобы повысить доступность приложений и служб. Он состоит из группы экземпляров независимых windows server с Microsoft служба отказоустойчивого кластера под управлением в этих экземплярах. Экземпляров windows server (или узлов как иногда называются) подключены, чтобы они могут взаимодействовать друг с другом, а также возможен сбой обнаружения. WSFC предоставляют сбоя функции обнаружения и переход на другой ресурс. Сбой узла или службы в кластере, затем обнаружил ошибку и передаются другому узлу автоматически или вручную для предоставления служб, размещенных на неисправного узла. Таким образом минимальный перебои в службах только пользователей и повысить доступность службы.

## <a name="sql-server-alwayson-availability-group"></a>Группа доступности Always On SQL Server

В этом разделе рассматриваются следующие задачи.

1.  [Включить группы доступности AlwaysOn SQL Server](#enable-sql-server-alwayson-availability-group-on-every-sql-server-instance).

2.  [Создание группы доступности](#create-an-availability-group).

3.  [Проверку и тестирование группы доступности](#validation-and-test-the-availability-group).

SQLServer AlwaysOn решения обеспечивают высокий уровень доступности и аварийного восстановления для баз данных SQLServer. AlwaysOn имеет два возможных решения.
Оба построены на основе WSFC.

-   Группы доступности AlwaysOn (AG)

-   Экземпляры AlwaysOn отказоустойчивого кластера (FCI).

Группы Доступности повышает доступность уровня базы данных. Группы Доступности (набор пользовательских баз данных) и его имя виртуальной сети регистрируются как ресурсы в кластере WSFC.

FCI повышает доступность уровня экземпляра. Службы SQL Server и связанных служб регистрируются как ресурсы в кластере WSFC. Кроме того решение отказоустойчивого Кластера требуется симметричное Общее дисковое хранилище, например в SAN или SMB общие файловые ресурсы, которые должны быть доступны всем узлам в кластере WFC.


### <a name="prerequisites"></a>Предварительные требования

-   Установка SQL Server на всех узлах. Дополнительные сведения см. в статье [Установка SQL Server 2016](https://docs.microsoft.com/sql/database-engine/install-windows/install-sql-server).

-   (Рекомендуется) Установите одинаковые наборы компонентов SQL Server и версии на каждом узле. В частности службы MDS должны быть установлены.

-   Подтвердите, что параметры брандмауэра Windows разрешает экземпляры SQL Server для взаимодействия друг с другом.

### <a name="create-an-availability-group"></a>Создание группы доступности

После включения функции AlwaysOn во всех экземплярах SQL Server, создания новой группы Доступности, содержащий базу данных MDS на одном узле.

Группы Доступности могут создаваться только на существующие базы данных. Таким образом, либо можно создать базу данных MDS на одном узле, или создайте временную базу данных, а затем удалите временной базы данных. В этом примере мы Создание emptyMDS базы данных и создание группы Доступности в этой базе данных MDS.

1.  Запустите **SQL Server Management Studio** (**SSMS**) на узле и подключитесь к локальному экземпляру SQL Server с соответствующими учетными данными.

2.  Откройте в среде SSMS **новый запрос** и выполните следующий скрипт для создания пустой базы данных. Замените C:\\temp с расположением будет использоваться для выполнения полной резервной копии.

    ```
    CREATE DATABASE MDS\_Sample
    ```

17.

    **Синхронная фиксация**: Это гарантирует, что если транзакция фиксируется в основной реплике базы данных, затем транзакция фиксируется на всех репликах синхронной. Это не гарантирует асинхронной фиксации и он может отставать от основной реплики.

    Синхронная фиксация обычно следует включать только в том случае, если два узла находятся в одном центре обработки данных. Если они находятся в разных центрах обработки данных, синхронной фиксации может снизить производительность базы данных.

    Если этот флажок не установлен, то будет использоваться асинхронной фиксации.

    **Автоматический переход на другой ресурс:** , если первичная реплика не работает, группой Доступности будет автоматически отработки отказа его вторичную реплику при выборе автоматического перехода на другой ресурс. Это можно включить только на репликах с синхронной фиксации.

    **Вторичная реплика для чтения:** по умолчанию, пользователи не смогут подключиться к любой вторичной репликой. Это позволит пользователям подключаться к вторичной реплике с доступом только для чтения.

## <a name="conclusion"></a>Заключение

В этом документе мы увидели, как установить и настроить серверной базе данных Master Data Services на основе группы доступности AlwaysOn SQL Server. Эта конфигурация обеспечивает высокий уровень доступности и аварийного восстановления в серверной базе данных Master Data Services. Чтобы реализовать эту конфигурацию, необходимо установить и настроить Windows Server отказоустойчивого кластера, AlwaysOn группы доступности SQL Server и Master Data Services.
