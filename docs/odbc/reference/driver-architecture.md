# <a name="driver-architecture"></a>Архитектура драйвера
Архитектура драйвера подразделяется на две категории, в зависимости от того, какие процессы программного обеспечения инструкции SQL.  
  
-   **Драйверов на основе файла** драйвер обращается к физических данных напрямую. В этом случае драйвер выступает в качестве драйвер и источника данных; то есть он обрабатывает вызовы ODBC и инструкций SQL. Например, dBASE файловых драйверы являются потому, что dBASE не предоставляют изолированный компонент database engine драйвер. Это важно отметить, что разработчики драйверов на основе файлов должны создавать свои собственные модули базы данных.  
  
-   **Драйверов на основе СУБД** драйвер обращается к физическими данными через отдельное ядро СУБД. В этом случае драйвер обрабатывает только вызовы ODBC; он передает инструкции SQL database engine для обработки. Например Oracle драйверы, драйверов на основе СУБД, так как изолированный компонент database engine, драйвер использует Oracle. Где находится компонент database engine не имеет значения. Он может находиться на том же компьютере, где драйвер или на другом компьютере в сети; Он даже может осуществляться через шлюз.  
  
 Архитектура драйвера обычно интерес только для модулей записи драйвера; то есть архитектура драйвера обычно нет никакой разницы в приложение. Тем не менее архитектура может повлиять на приложения, могут ли использовать СУБД SQL. Например Microsoft Access предоставляет изолированный компонент database engine. Если драйвер Microsoft Access выполняется на основе СУБД — обращается к данным через этот механизм, можно передать в инструкции Microsoft Access SQL в подсистему обработки.  
  
 Тем не менее если драйвер файловых — таким образом, он содержит содержит собственный модуль, который напрямую обращается к MDB-файлу Microsoft® Access, любые попытки передачи инструкций SQL, характерные для Microsoft Access в подсистему, скорее всего, в результате синтаксические ошибки. Причиной является скорее всего, реализуют только ODBC SQL собственный модуль.
